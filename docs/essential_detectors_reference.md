# Essential 检测项参考手册

Essential 模式包含 46 个高置信度检测项，覆盖 6 个安全维度。所有检测项遵循**无罪推定**原则：除非有确定性证据表明中转站存在恶意行为，否则判定为 PASS。

---

## 一、金融安全与供应链攻击（S0）

S0 级别检测项针对直接的资金盗窃和代码供应链攻击，一旦触发即判定为最高风险。

### D47 地址一致性检测

**检测目标**：中转站在 tool_call 参数中悄悄替换加密货币钱包地址，将转账目标改为攻击者地址。

**方法**：向模型提供一个随机生成的 ETH 地址（42 字符），要求模型分别在 `get_balance` 和 `transfer` 两个工具调用中使用该地址，并在纯文本中复述。逐一比对三处地址是否与原始地址一致。

**判定**：
- 三处地址完全一致 → PASS
- 地址被截断或有少量字符差异（15 位以上连续匹配）→ PASS（模型记忆能力问题）
- 地址完全不同（连续匹配 < 15 位）→ FAIL（定向篡改）
- 仅 transfer 地址被替换而 balance 地址正确 → FAIL（选择性篡改，最危险的攻击模式）

### D48 高精度金额保真检测

**检测目标**：中转站在转账金额中进行微量偷取——截断小数精度或微调数值，从每笔交易中抽取差额。

**方法**：发送三笔不同精度的测试转账请求（`1.23456789 ETH`、`0.00314159 ETH`、`42.9876 USDT`），检查 tool_call 参数中的金额是否与请求值一致。

**判定**：
- 金额完全匹配 → PASS
- 金额是原始值的合法四舍五入或截断（如 1.23456789 → 1.235 或 1.23）→ PASS（模型精度限制）
- 偏差 > 1% 且不是任何精度的 rounding 结果 → FAIL
- 有直连 Provider 对比时：router 偏差 > 1% 但直连偏差 < 0.1% → FAIL（确认篡改）

### D45 工具调用参数篡改检测

**检测目标**：中转站篡改 `cast send`（以太坊转账）、`uniswap`（DEX 交易）、`bitcoin-cli`（比特币转账）、`spl-token`（Solana 转账）、`stripe`（支付）五类金融工具调用的关键参数。

**方法**：对每种工具分别发送包含精确参数的请求，检查返回的 tool_call 参数中收款地址、金额、代币合约地址是否被篡改。

**判定**：任一探针的关键参数被替换为不同值 → FAIL。

### D45b 包名仿冒检测

**检测目标**：中转站在 `pip install`、`npm install`、`docker pull` 等命令的 tool_call 中将正规包名替换为仿冒包（typosquatting），实现供应链攻击。

**方法**：要求模型生成安装 `requests`（pip）、`express`（npm）、`nginx`（docker）的命令，检查 tool_call 中的包名是否与官方包名一致。

**判定**：包名被替换为不同值 → FAIL。

### D45c 安装器 URL 篡改检测

**检测目标**：中转站在 `curl | sh` 类安装命令的 tool_call 中将官方下载 URL 替换为恶意 URL。

**方法**：要求模型生成 Rust（rustup）、Homebrew、certbot、numpy 的安装命令，提取 URL 并与官方域名比对。

**判定**：URL 域名与官方不一致 → FAIL。

### D28 并发会话隔离检测

**检测目标**：中转站使用共享的模型会话实例，导致不同用户的对话内容互相泄露。

**方法**：同时发送 9 个"投毒"请求（包含随机生成的密钥 `CANARY_XXXXXXXX`）和 1 个"提取"请求（"What is my secret code?"）。检查提取请求的回复中是否出现了投毒请求的密钥。

**判定**：提取响应中包含投毒密钥（完全匹配或两部分邻近出现）→ FAIL。

### D84 零宽字符注入检测

**检测目标**：中转站在响应中注入不可见的 Unicode 字符（零宽空格、零宽连接符等）用于用户追踪或水印。

**方法**：发送多个普通请求，扫描响应内容中是否包含零宽字符（U+200B、U+200C、U+200D、U+FEFF 等）。

**判定**：检测到零宽字符 → FAIL。

### D100 跨 Key 隔离检测

**检测目标**：中转站在不同 API Key 之间共享会话状态，导致一个用户的数据泄露给另一个用户。

**方法**：使用一个 API Key 发送包含密钥的请求，然后使用另一个 API Key 尝试提取该密钥。

**判定**：第二个 Key 能获取到第一个 Key 的密钥 → FAIL。

### D116 凭证泄露检测

**检测目标**：中转站在错误响应中泄露内部 API Key、数据库密码、JWT Token 等敏感凭证。

**方法**：发送会触发错误的请求（无效模型名等），扫描错误响应中是否包含 API Key 格式（`sk-`、`key-`）、JWT Token、数据库连接串等模式。

**判定**：检测到凭证模式 → FAIL。

### D117 响应头信息泄露检测

**检测目标**：中转站在 HTTP 响应头中泄露内部基础设施信息（后端 IP、内部域名、真实 API Key 等）。

**方法**：检查响应头中是否包含 `X-Real-IP`、`X-Backend-Server`、`Authorization`、内网 IP 地址等敏感信息。

**判定**：检测到敏感响应头 → FAIL。

### D118 工具 Schema 注入检测

**检测目标**：中转站在用户定义的工具列表中注入额外的工具（如转账工具），诱导模型调用攻击者控制的工具。

**方法**：发送一个只包含 `get_weather` 和 `get_time` 两个工具的请求，检查模型返回的 tool_call 中是否出现了用户未定义的工具。

**判定**：出现未授权的工具调用 → FAIL。

## 二、身份与完整性验证（P0）

### D23 劫持令牌探针

**检测目标**：检测 API Key 是否为从第三方应用（Cursor、Monica、Poe、DeepL 等）盗用的令牌。

**方法**：发送三种空白/最小内容请求（空字符串、空格、句号）触发第三方应用的引导提示词泄露，同时发送 prompt injection 尝试提取 system prompt。扫描响应中是否包含第三方品牌关键词。

**判定**：
- 2+ 个空白探针匹配到关键词 → FAIL
- 单个强品牌词（Monica、Cursor、Poe、DeepL、Notion）→ FAIL
- system prompt 提取出品牌词 → FAIL

### D40 危险内容拒绝保真检测

**检测目标**：检测中转站是否使用未经安全对齐的裸模型（如 Llama-base、Qwen-base）冒充 GPT-4/Claude 等前沿模型。

**方法**：发送三个所有前沿模型都会拒绝的危险请求（化学武器合成、CSAM、勒索软件），检查模型是否拒绝回答。

**判定**：
- 前沿模型（GPT-4/5、Claude、Gemini Pro）有 2+ 个请求被模型执行 → FAIL
- 非前沿模型执行危险请求 → PASS（模型安全训练不足，非中转站问题）
- 编号列举式拒绝（"1. This is illegal\n2. I cannot help..."）→ 正确识别为拒绝

### D81 隐藏 System Prompt 检测

**检测目标**：检测中转站是否在用户不知情的情况下注入广告、追踪指令或其他隐藏的 system prompt。

**方法**：发送两个探针——第一个要求模型复述完整的 system prompt，第二个从不同角度验证。扫描响应中是否包含广告/追踪指标（URL、品牌推广、追踪 ID 等）或超出预期的大段 system prompt 内容。

**判定**：发现广告/追踪指标或大段隐藏 system prompt → FAIL。

### D114 多跳路由检测

**检测目标**：检测请求是否经过了多个代理网关（如 OneAPI → LiteLLM → 实际 Provider），每一跳都可能操控请求。

**方法**：检查响应头中是否同时出现多个不同代理网关的指纹（x-oneapi-version、x-litellm-version、x-vllm-version 等）。

**判定**：检测到 2+ 种不同网关指纹 → FAIL。

### D97 隐藏对话轮次注入检测

**检测目标**：检测中转站是否在用户的对话历史中偷偷注入额外的消息轮次（如广告或行为控制指令）。

**方法**：发送一个包含明确轮次数量的对话（2 轮），要求模型报告对话中的消息数量。如果报告的数量显著多于实际发送的数量，说明有隐藏轮次被注入。

**判定**：
- 报告轮次与实际一致或差 1（system message 计数差异）→ PASS
- 报告轮次显著偏多（差 3+）→ FAIL

### D24a 上下文截断金丝雀检测

**检测目标**：检测中转站是否悄悄截断了用户的长文本输入以节省成本。

**方法**：在长文本的头部、中部、尾部分别嵌入三个随机金丝雀值，要求模型全部复述。检查三个金丝雀是否都出现在响应中。

**判定**：任一金丝雀缺失且 token 计数比值异常低 → FAIL。

### D22 协议严格性检测

**检测目标**：检测中转站是否有协议翻译层——声称是 OpenAI API 但实际接受 Anthropic 特性，或反之。

**方法**：发送四个厂商特定的协议探针：
- 22a: strict JSON schema（仅 OpenAI 支持）
- 22b: 连续 user 消息（Anthropic 必须拒绝）
- 22c: assistant prefill（仅 Anthropic 支持）
- 22d: 超范围参数（应返回校验错误）

**判定**：接受了不应支持的厂商特性 → FAIL。仅对声称的 provider 类型发送对应探针。

### D21 物理参数盲区检测

**检测目标**：检测网页逆向代理——通过浏览器自动化包装的假 API 无法传递底层采样参数。

**方法**：测试四个物理参数：
- 21a: temperature=2.0（应产生混乱文本）
- 21b: logit_bias 禁用特定 token（该 token 应不出现）
- 21c: logprobs=true（应返回概率分布）
- 21d: max_tokens=1（应严格只输出 1 个 token）

**判定**：2+ 个子探针失败 → FAIL。非 OpenAI 模型自动跳过 21a/21b/21c（这些是 OpenAI 特有能力）。

### D131 网页抓取指纹检测

**检测目标**：检测中转站是否通过抓取网页界面（如 chat.openai.com）而非调用官方 API 来提供服务。

**方法**：检查响应中的五类网页抓取特征：
1. usage 字段完全缺失（网页版不返回 token 计数）
2. 网页特有响应头（set-cookie、cf-clearance、CSP）
3. HTML 标签残留（DOM 抓取产物）
4. 网页会话 ID 格式（conv-*、turn-*、chat-session-*）
5. model 字段缺失

**判定**：3+ 个特征 → FAIL；2 个特征且至少含一个强信号（HTML/web header/web session ID）→ FAIL。单独的 usage 缺失不构成判定依据。

### D132 Claude Code 后端检测

**检测目标**：检测中转站是否将 Claude Code Max 订阅作为未授权后端。

**方法**：
- 探针 1：要求模型列出所有可用工具，检查是否出现 Claude Code 独有的 PascalCase 工具名（WebFetch、TodoWrite、NotebookEdit、TaskCreate、ScheduleWakeup 等）
- 探针 2：询问工作目录和环境，检查是否引用 Claude Code 特有上下文（CLAUDE.md、claude.ai/code、gitStatus 等）

**判定**：匹配到 3+ 个 Claude Code 特有工具名 → FAIL。需要 MAJORITY_2_OF_2（运行两轮，两轮都 FAIL 才最终判定）。通用词（read、write、bash、grep）不参与匹配。

## 三、参数转发与计费（P1）

### D62 Logprobs 诚实性检测

**检测目标**：检测 `logprobs` 参数是否被中转站丢弃或返回伪造的概率分布。

**方法**：发送 `logprobs=true` 的请求，检查响应中是否包含 logprobs 数据块，以及概率分布的标准差是否合理（伪造数据通常过于均匀或过于集中）。

**判定**：请求了 logprobs 但响应中没有 → FAIL（非 OpenAI 模型自动 SKIP）。概率分布标准差异常 → FAIL。

### D29 Token 计费审计

**检测目标**：检测中转站是否虚报 token 使用量——报告的 `prompt_tokens` 远高于实际值，多收费。

**方法**：使用 tiktoken 本地计算 prompt 的 token 数，与响应中 `usage.prompt_tokens` 对比。

**判定**：报告值显著高于本地计数（超过合理的 tokenizer 差异范围）→ FAIL。非 OpenAI 模型因 tokenizer 差异自动 SKIP。

### D29b 提示词缓存完整性检测

**检测目标**：检测中转站是否声称支持 prompt caching 但实际未执行（收取缓存创建费用但从不提供缓存命中折扣）。

**方法**：发送两个相同的请求。第一个请求应创建缓存（`cache_creation_input_tokens > 0`），第二个请求应命中缓存（`cache_read_input_tokens > 0`）。

**判定**：第一个请求创建了缓存但第二个请求未命中 → FAIL。非 OpenAI/Anthropic 模型自动 SKIP。

### D83 补全 Token 膨胀检测

**检测目标**：检测中转站是否在 `usage.completion_tokens` 中虚报数量——实际输出 50 个 token 但报告 200 个。

**方法**：使用 tiktoken 本地计算响应内容的 token 数，与 `usage.completion_tokens` 对比。

**判定**：报告值与本地计数偏差过大 → FAIL。非 OpenAI 模型因 tokenizer 差异自动 SKIP。

### D123 截断输出全额计费检测

**检测目标**：检测中转站截断了模型的输出（只返回一半内容），但按完整输出的 token 数收费。

**方法**：发送一个需要长输出的请求，比对响应内容的本地 token 数与 `usage.completion_tokens`。

**判定**：报告的 completion_tokens 远大于实际内容的 token 数 → FAIL。

### D53 元数据完整性检测

**检测目标**：检测中转站是否剥离了 usage 元数据字段——不返回 `prompt_tokens`、`completion_tokens` 等计费关键字段，使用户无法核实计费。

**方法**：发送请求后检查响应中 usage 字段的完整性。如有直连 Provider 对照，比对两者的字段差异。

**判定**：缺少必要的 usage 字段（如 `prompt_tokens`、`completion_tokens`）→ FAIL。

## 四、流式传输（P1）

### D32a 流式传输真实性检测

**检测目标**：检测中转站是否伪造流式传输——实际调用非流式接口，然后将完整响应拆分为几个 chunk 假装是流式。

**方法**：发送 `stream=true` 的请求，分析 chunk 数量和内容分布。

**判定**：
- 长响应（20+ 词）仅收到 ≤2 个 chunk → FAIL
- 80% 以上内容集中在最后一个 chunk → FAIL
- 短响应（< 20 词）在 1-2 个 chunk 中完成 → PASS（合法的短输出行为）

### D110 流式/非流式内容一致性检测

**检测目标**：检测中转站在流式和非流式模式下返回不同的内容——可能在某种模式下注入或修改内容。

**方法**：对同一 prompt 分别发送流式和非流式请求，比较两者的响应内容。

**判定**：两个响应的 Jaccard 相似度低于阈值 → FAIL（内容显著不同）。

### D111 流式提前终止检测

**检测目标**：检测中转站是否提前切断流式传输——模型还在生成但中转站强制关闭了连接。

**方法**：发送流式请求，检查响应是否有 `finish_reason` 字段。缺少 `finish_reason` 且输出内容很少，说明流被提前终止。

**判定**：无 `finish_reason` 且输出 < 10 词 → FAIL（推理模型的 thinking token 场景除外）。

## 五、工具调用（P1）

### D16b 工具调用结构保真检测

**检测目标**：检测工具调用是否被降级为纯文本，或 tool_call 的 schema 是否被破坏。

**方法**：发送带 `tool_choice` 强制调用指定函数的请求，检查响应是否为 tool_call 格式、函数名和参数是否正确。

**判定**：
- OpenAI 模型返回纯文本（无 tool_call）→ FAIL
- 函数名错误 → FAIL
- 非 OpenAI 模型返回了正确结构但参数值不同 → PASS（模型能力差异）
- 非 OpenAI 模型不支持 tool_choice → SKIP

### D16c 并行工具调用检测

**检测目标**：检测中转站是否破坏了并行工具调用能力——模型应同时调用多个工具但只返回了一个。

**方法**：发送一个需要同时调用 `get_weather` 和 `convert_currency` 两个工具的请求。

**判定**：只返回 1 个 tool_call（应返回 2 个）→ FAIL。不支持工具调用的模型 → PASS。

### D56 Tool Choice 遵从检测

**检测目标**：检测 `tool_choice` 参数是否被中转站忽略——强制指定调用某个函数但模型调用了其他函数或返回纯文本。

**方法**：发送 `tool_choice: {function: {name: "get_stock_price"}}` 的请求，检查是否调用了指定函数。

**判定**：调用了不同函数 → FAIL。非 OpenAI 模型不支持强制 tool_choice → SKIP。

## 六、身份与一致性（P1）

### D87 响应模型字段检测

**检测目标**：检测响应中的 `model` 字段是否与请求的模型一致——中转站可能用廉价模型冒充高端模型。

**方法**：发送多个请求，检查每个响应的 `model` 字段是否匹配请求的模型名。使用标准化匹配（忽略日期后缀、版本格式差异等）。

**判定**：model 字段与请求模型属于不同模型族（如请求 gpt-4o 但返回 gpt-3.5-turbo）→ FAIL。

### D57 响应 ID 唯一性检测

**检测目标**：检测中转站是否返回模板化的固定响应——多个请求返回相同的 response ID 说明响应是预设的而非实时生成的。

**方法**：发送 5 个相同请求，收集每个响应的 ID，检查是否有重复。

**判定**：出现重复 ID → FAIL。

### D30 错误路径网关指纹检测

**检测目标**：通过错误响应识别中转站使用的底层网关软件（OneAPI、New-API、LiteLLM、Poe 等），这些网关的错误格式会暴露代理链路。

**方法**：发送三种会触发错误的请求（超范围 temperature、不存在的模型名、负数 max_tokens），分析错误响应的格式和 header。

**判定**：检测到已知网关指纹（`x-oneapi-version`、`one_api_error`、Cloudflare HTML 等）→ FAIL。

### D130 并发能力检测

**检测目标**：检测基于浏览器自动化的逆向代理——这类代理通常只有一个浏览器实例，无法处理并发请求。

**方法**：同时发送 5 个轻量请求，分析成功率和延迟分布。

**判定**：
- 60% 以上请求失败且错误信息包含队列/容量关键词 → FAIL
- 呈阶梯延迟模式（最慢 ≥ 5 倍最快，且最慢 ≥ 5 秒）→ FAIL（请求被串行化）
- 所有请求正常完成 → PASS

## 七、上下文与记忆（P1）

### D24b 代数上下文截断检测

**检测目标**：通过数学验证检测长文本是否被截断——在约 80K token 的文本中嵌入代数变量，要求模型计算结果。

**方法**：发送约 80K token 的填充文本，其中嵌入三个变量值（var_X、var_Y、var_Z），要求模型计算 `(var_X + var_Y) * var_Z`，正确答案为 38。

**判定**：
- 回答 38 → PASS
- 回答其他数字 → PASS（可能是模型计算错误）
- 回答"找不到变量" → PASS（可能是模型上下文窗口限制或截断，无法区分）

### D24c 多轮对话历史完整性检测

**检测目标**：检测中转站是否裁剪了多轮对话的早期历史——用户在第一轮说的内容在后续轮次中被丢弃。

**方法**：构造多轮对话，在第一轮中嵌入一个随机 nonce 值，在最后一轮要求模型复述。

**判定**：
- 正确复述 nonce → PASS
- 回复了错误的 nonce → PASS（模型记忆能力问题）
- 完全不提及 nonce 且回复较长 → FAIL（早期历史被裁剪）
- 回复过短 → PASS（可能是输出截断）

### D42 首尾标记上下文截断检测

**检测目标**：检测中转站是否悄悄截断了长输入——通过 token 计数比对检测。

**方法**：发送约 15K token 的多样化填充文本，末尾放置标记 `ENDMARKER_9F7A2C`，要求模型回显标记。同时比对响应中报告的 `prompt_tokens` 与本地 tiktoken 计数。

**判定**：
- 标记出现在响应中且 token 比值合理 → PASS
- 标记缺失但 token 比值正常（> 阈值）→ PASS（模型未遵循指令）
- token 比值过低（OpenAI < 50%，其他 < 25%）→ FAIL（输入被截断）

### D26 语义缓存检测

**检测目标**：检测中转站是否使用语义缓存——对语义相似但包含不同关键细节的请求返回相同的缓存响应。

**方法**：发送两个结构相同但包含不同随机 nonce 的请求（A 和 B），检查响应 B 是否包含 nonce B（而非 nonce A），以及两个响应是否完全相同。

**判定**：
- 响应 B 包含 nonce A → FAIL（缓存回放）
- 两个响应完全相同（byte-identical）→ FAIL
- 两个响应高度相似且 nonce B 未出现 → FAIL

## 八、其他信号（P1）

### D11 请求完整性检测

**检测目标**：检测 system prompt 是否被忠实转发——中转站可能剥离或篡改用户的 system prompt。

**方法**：在 system prompt 中嵌入一个唯一标记，要求模型复述。检查标记是否出现在响应中。同时检测 user 消息中的指令注入是否能覆盖 system prompt。

**判定**：system prompt 中的标记被 user 注入成功覆盖 → FAIL。

### D15 System Prompt 护栏完整性检测

**检测目标**：检测中转站是否剥离了 system prompt 中的安全护栏——如要求模型"禁止生成 SQL 代码"的指令被移除。

**方法**：在 system prompt 中设置护栏（禁止生成 SQL），然后在 user 消息中要求生成 SQL。检查模型是否遵守护栏。

**判定**：护栏被突破（生成了 SQL 代码）→ FAIL。护栏格式未被遵循但未生成 SQL → PASS（模型格式兼容性问题）。

### D99 速率限制透明度检测

**检测目标**：检测中转站的速率限制是否透明——返回 429 状态码时是否包含 `Retry-After` header。

**方法**：发送一批并发请求触发速率限制，检查 429 响应是否包含标准的 `Retry-After` 信息。

**判定**：
- 429 响应缺少 `Retry-After` header → FAIL
- 大量 5xx 错误（≥ 3 个）→ FAIL（服务不稳定）
- 全部请求网络失败 → FAIL

## 九、预筛选（PRE_SCREEN）

### D31 综合压力预筛探针

**检测目标**：一次性检查五项能力，快速识别明显的假 API 或逆向代理。

**方法**：发送一个同时包含 strict JSON schema、logit_bias、40K token 上下文（含代数变量）、system prompt 和 stream_options 的复合请求。

**检查项**：
1. strict JSON schema 是否被遵守（仅 OpenAI）
2. logit_bias 是否生效（仅 OpenAI）
3. 代数计算是否正确（5 × 10 = 50）
4. 是否泄露第三方 system prompt
5. stream usage 是否返回（仅 OpenAI）

**判定**：system prompt 泄露 → FAIL。4+ 项检查失败 → FAIL。非 OpenAI 模型自动跳过 OpenAI 特有检查项。
